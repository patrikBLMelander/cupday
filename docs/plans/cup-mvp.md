# Cup MVP â€” Implementation Plan

Implementation plan for `docs/tasks/cup-mvp.md`. Covers project structure, tech wiring, route map, mock API shape, theming, and a recommended task sequence for `/implement-frontend`.

## High-Level Strategy

1. **Scaffold once, by hand.** The initial Vite + tooling setup is one-shot, not a feature â€” done outside `/implement-frontend`.
2. **Build admin before public.** You need to be able to create a cup before there's anything for visitors to see.
3. **Mock all API calls through MSW.** Handlers read/write an in-memory DB persisted to `localStorage`. When the Spring backend lands, swap the RTK Query `baseUrl` and delete the MSW setup.
4. **Theme everything via CSS custom properties from day one** â€” even pages without a per-cup theme use the same vars with default values, so adding theming later is just changing the values.
5. **Each feature task is small enough to fit `/implement-frontend` cleanly** â€” single-flow, test-coverable, ships independently.

## State & Data

### RTK Query slices

One slice per feature. All target the same `baseUrl: '/api'` so swapping to a real backend is one config line.

- **`authApi`** â€” `login`, `logout`, `getMe`
- **`cupsApi`** â€” `listCups`, `getCup`, `getCupBySlug`, `createCup`, `updateCup`, `deleteCup`
- **`teamsApi`** â€” `listTeamsByCup`, `createRegistration` (public), `getRegistration` (public), `updateTeamStatus`, `assignTeamGroup`, `cancelTeam`
- **`scheduleApi`** â€” `listMatchesByCup` (public), `generateSchedule`, `updateMatch`

Local UI state stays in component state or a small `uiSlice` (e.g., language preference, admin filters) â€” RTK Query owns server state.

## Mock API (MSW handlers)

All endpoints under `/api`. JSON bodies. Spring-friendly conventions: camelCase keys, ISO-8601 dates, RFC-7807 problem details for 4xx errors.

### Auth
- `POST /api/auth/login` â€” `{ email, password }` â†’ `{ token, user }` or 401
- `POST /api/auth/logout` â†’ 204
- `GET /api/auth/me` â†’ `{ user }` (Bearer token in `Authorization`)

### Cups (admin, requires auth)
- `GET /api/admin/cups` â†’ `Cup[]`
- `POST /api/admin/cups` â†’ `Cup`
- `GET /api/admin/cups/{id}` â†’ `Cup`
- `PATCH /api/admin/cups/{id}` â†’ `Cup`
- `DELETE /api/admin/cups/{id}` â†’ 204

### Cups (public)
- `GET /api/cups/by-slug/{slug}` â†’ `Cup` (omits private fields like internal notes)

### Teams
- `GET /api/admin/cups/{id}/teams` â†’ `Team[]` (all statuses, requires auth)
- `POST /api/cups/{id}/registrations` (public) â€” `{ clubName, contactName, contactEmail, contactPhone, teamNames: string[] }` â†’ `{ registrationId, teamIds }` or 409 with the conflicting team name
- `GET /api/registrations/{id}` (public) â†’ `{ registration, teams, cup }` for the payment page
- `PATCH /api/admin/teams/{id}` (auth) â€” `{ status?, groupLabel? }` â†’ `Team`
- `DELETE /api/admin/teams/{id}` (auth) â†’ 204

### Schedule
- `GET /api/cups/{id}/matches` (public) â†’ `Match[]`
- `POST /api/admin/cups/{id}/schedule/generate` (auth) â€” `{ startTime, matchDurationMinutes, breakBetweenMatchesMinutes }` â†’ `Match[]`
- `PATCH /api/admin/matches/{id}` (auth) â€” `{ pitch?, startTime?, homeTeamId?, awayTeamId? }` â†’ `Match`

## Schedule Generator Logic

Pure function in `features/schedule/scheduleGenerator.ts`. Deterministic, unit-tested.

Inputs:
- `groupATeams: Team[4]`, `groupBTeams: Team[4]`
- `pitchCount: 2`
- `startTime: Date`, `matchDurationMinutes`, `breakBetweenMatchesMinutes`

Output: `Match[]` (12 matches: 6 per group, round-robin).

Algorithm:
1. Compute round-robin pairings per group using the standard circle method (3 rounds Ă— 2 matches per round per group = 6 matches per group).
2. Interleave so round R of group A plays on pitch 1 while round R of group B plays on pitch 2. This avoids two teams from the same group playing simultaneously and uses both pitches every slot.
3. Total slots = 6 (3 group-A rounds + 3 group-B rounds â€¦ wait â€” both pitches run at once, so 6 rounds means 6 time slots if A and B share slots, or 3 slots if they alternate).
4. **Picked layout:** 6 time slots. Slot N: pitch 1 = group A round (N mod 3), pitch 2 = group B round (N mod 3). After 3 slots both groups have played all their rounds â€” but each round contains 2 matches per group, so we need 6 slots per group â†’ 6 slots total with both pitches busy each slot. Confirmed: 6 slots Ă— 2 pitches = 12 matches.

Generator produces matches with `startTime + (slotIndex * (duration + break))`.

## Auth (Mock Strategy)

- **Login form** accepts any email + password â‰Ą 6 chars. Returns a fake bearer token.
- Token stored in `localStorage` under `cup.auth.token`.
- RTK Query `prepareHeaders` injects `Authorization: Bearer <token>`.
- `RequireAuth` route guard checks `getMe` â†’ redirects to `/admin/login` on 401.
- `mocks/handlers.ts` validates the token shape and returns a hardcoded user.

When the backend lands: replace handlers with real Spring Security endpoints; everything else stays the same.

## Testing Strategy

- **Unit:** `scheduleGenerator` (deterministic, easy), `lib/slug.ts`, `lib/hexToHslChannels.ts`.
- **Component:** Forms (registration name conflict, login validation), layouts (theming applies, language switcher works).
- **Integration (MSW node server):** registration flow (POST registration â†’ GET registration on payment page), admin status transitions, schedule generation end-to-end.
- **E2E (Playwright):** Skip for MVP. Add when there's a deployed Railway instance to point at.

Tests are proportional. Don't write 7 tests for a 30-line component.

## Scaffold Step (manual, before Task 1)

This is too cross-cutting for `/implement-frontend` and only happens once. I'd run it as a single conversation step:

1. `npm create vite@latest . -- --template react-ts`
2. Install deps: `react-router-dom`, `@reduxjs/toolkit`, `react-redux`, `react-i18next`, `i18next`, `i18next-browser-languagedetector`, `tailwindcss`, `postcss`, `autoprefixer`, `class-variance-authority`, `clsx`, `tailwind-merge`, `msw`
3. `npx tailwindcss init -p`
4. `npx shadcn@latest init` (configure to use CSS vars)
5. `npx msw init public/ --save`
6. Create the folder structure above
7. Wire `main.tsx` â†’ MSW worker start â†’ Redux Provider â†’ Router
8. Drop in CSS-var theme defaults, language detector, and a placeholder home page
9. Verify `npm run dev` shows the placeholder, `npm run test:once` passes (one trivial test), `npm run lint` passes

End state: a working dev server, MSW intercepting one trivial endpoint, theme vars in place, i18n with one `t('hello')` string in both languages.

## Risks & Notes

- **shadcn/ui** is a CLI generator, not a runtime package. Each component is added with `npx shadcn@latest add <name>`. Plan to add components incrementally per task (button, input, dialog, card, dropdown, select, tabs, toast).
- **MSW + Vite** requires `public/mockServiceWorker.js` and a conditional `worker.start()` in dev only. Easy to mess up â€” verify the network tab shows MSW intercepting on first dev run.
- **Color contrast** â€” the per-cup primary may have poor contrast against white text. For MVP, accept user-chosen colors and note this as a v2 polish item (compute foreground from luminance). Provide a small set of preset color pairs (e.g., 6 common Swedish club color combos) as a fallback to nudge good choices.
- **Slug uniqueness** across cups â€” generate from name + a short random suffix to keep it readable but collision-free.
- **localStorage size** for the mock DB is trivial at MVP scale (~10 KB). No concern.
- **`maxTeams` adjustment after registrations exist** â€” if the organizer lowers `maxTeams` below current count, the UI should warn and refuse. Edge case but cheap to handle.
- **Cup `status` transitions** are mostly automatic (`open â†’ full` on registration count, `scheduled` when schedule is locked) but `draft â†’ open` is a manual organizer action ("Open registrations"). Make this explicit in `AdminCupSettingsPage`.
